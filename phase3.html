// pages/api/auth/[...nextauth].ts // NextAuth configuration — providers, Prisma adapter, session callbacks. // authOptions is exported so getServerSession() can reuse it in any route. import NextAuth, { type AuthOptions } from 'next-auth' import GithubProvider from 'next-auth/providers/github' import GoogleProvider from 'next-auth/providers/google' import { PrismaAdapter } from '@auth/prisma-adapter' import { prisma } from '@/lib/prisma' export const authOptions: AuthOptions = { adapter: PrismaAdapter(prisma), providers: [ GithubProvider({ clientId: process.env.GITHUB_CLIENT_ID!, clientSecret: process.env.GITHUB_CLIENT_SECRET!, }), GoogleProvider({ clientId: process.env.GOOGLE_CLIENT_ID!, clientSecret: process.env.GOOGLE_CLIENT_SECRET!, }), ], session: { // Sessions stored in the DB (via Prisma adapter), not as JWTs. // This lets us revoke sessions server-side if needed. strategy: 'database', }, callbacks: { // Attach user.id to every session object. // Without this, session.user only has email/name/image from OAuth. async session({ session, user }) { if (session.user) { session.user.id = user.id } return session }, // Auto-generate a username on first OAuth sign-in. // Subsequent sign-ins skip this — username already exists. async signIn({ user }) { if (!user.email) return false const existing = await prisma.user.findUnique({ where: { email: user.email }, select: { username: true }, }) if (!existing?.username) { // Build from display name → strip non-alphanum → suffix with 4 random digits const base = (user.name ?? user.email.split('@')[0]) .toLowerCase() .replace(/[^a-z0-9]/g, '') .slice(0, 15) const username = `${base}${Math.floor(Math.random() * 9000) + 1000}` await prisma.user.update({ where: { email: user.email }, data: { username }, }) } return true }, }, pages: { signIn: '/auth/signin', error: '/auth/error', }, } export default NextAuth(authOptions)

'...' : 'submit'} </button> </form> {wrongResult && ( <div className="attempt-feedback"> <span className="wrong-label">✗ incorrect</span> {attemptsLeft !== null && ( <span className="attempts-left">{attemptsLeft} attempts remaining</span> )} {wrongResult.hint && ( <div className="hint-reveal">💡 {wrongResult.hint}</div> )} </div> )} {puzzle.maxAttempts && attemptsUsed > 0 && ( <div className="attempt-bar"> <div className="attempt-bar-fill" style={{ width: `${(attemptsUsed / puzzle.maxAttempts) * 100}%` }} /> </div> )} </div> ) }

NOTES auth/[...nextauth].ts
NextAuth Configuration

Two OAuth providers (GitHub + Google) with the Prisma adapter. Sessions are stored in the database, not as JWTs — this lets you revoke sessions server-side and attach any user data to the session object.

The session callback attaches user.id to every session. Without it, the default NextAuth session has no ID — just email, name, image. Every API route in Phase 2 depends on session.user.id.

The signIn callback auto-generates a username on first OAuth sign-in. It builds it from the display name, strips non-alphanumeric chars, and appends 4 random digits to avoid collisions.

⚠️ authOptions is exported so getServerSession(req, res, authOptions) can be called in any API route. Never call it without passing authOptions — it'll return null every time.
Feed Page

Server-side renders the session (so the page knows if you're logged in on first load), then uses SWR for client-side data fetching so sorting and filtering feel instant.

keepPreviousData: true in SWR means the old results stay visible while the new ones load — no jarring blank state on every sort change.

Tag filtering works by appending &tag=mystery to the SWR key. Changing the key triggers a refetch automatically. Clicking ✕ on the active tag clears it and the key changes back.

✓ Skeleton cards (9 divs with shimmer CSS) show while loading so layout doesn't shift when the real cards arrive.
Rabbit Hole Page ⚡

This is the entire game experience in one component. It loads the hole data (all levels, unlock status), renders the nav sidebar and current level, and handles the solve event.

On a correct answer: reloadHole() fires immediately to get the unlocked level content from the server, then after 1.4s the sidebar auto-advances to the next level. This gives the XPToast time to be visible before navigation.

Locked levels show only the puzzle question (no story content). The API enforces this — content is null in the response for locked levels. The UI reflects exactly what the server sends.

✓ The level sidebar is purely decorative for locked levels (button is disabled). You can see the depth of the hole but can't jump ahead.
Profile Page

Fully server-side rendered — no client data fetching. Profile data doesn't change often, so SSR is appropriate here. The XP progress bar percentage is calculated server-side too.

The "holes solved" count uses a raw SQL query (same pattern as the badges engine) to count distinct threads where the user unlocked the final level. Prisma's fluent API would need multiple queries to accomplish this.

Badge cards are colour-coded by rarity using RARITY_COLORS — COMMON is grey, RARE is cyan, LEGENDARY is amber. Border colour only, so the card remains legible at all rarities.

Create Wizard

Three-step wizard: setup → levels → review. Each step validates before allowing advancement. The canAdvance object makes disabled state declarative rather than scattered across multiple handlers.

Levels are added to the DB sequentially in a for...of loop (not Promise.all) because level order matters — the API assigns levelNumber as totalLevels + 1 per call.

useSession({ required: true }) automatically redirects unauthenticated users to the sign-in page. Combined with the Edge middleware, the create route is protected at two layers.

⚠️ The plaintext answer field travels from creator → POST body → lib/answer.hashAnswer() → DB as a hash. It's never logged. The network layer is the only exposure window (use HTTPS).
NavBar

Reads session entirely client-side via useSession(). The three states — loading skeleton, unauthenticated sign-in button, authenticated user info — cover every case cleanly.

No prop drilling required. Any component tree can render NavBar and it will read its own session. This works because next-auth wraps the app in a SessionProvider in _app.tsx.
ThreadCard

The depth indicator (↓ N) uses colour to immediately convey difficulty: green for shallow (<5 levels), amber for moderate, red for deep (10+). Solvers can self-select based on commitment.

Tag clicks call e.preventDefault() before onTagClick(tag). Without the preventDefault, the Link's href would also fire, navigating away from the feed.

The rank color lookup uses the author's rank, not the viewer's. Seeing a "RABBIT MASTER" badge on an author signals that the content is probably high quality — social proof.

PuzzleBox

The shake animation fires on wrong answers — the input briefly oscillates left-right, giving haptic-style feedback that the answer was wrong without needing text. After shaking, the input clears and re-focuses.

The attempt bar fills left-to-right as wrong answers accumulate. When it hits 100%, the next attempt route returns 429. The bar makes the limit feel tangible rather than abstract.

The already-solved state renders a static "✓ solved" badge instead of the input. A user who revisits a solved level sees confirmation, not an empty puzzle box.

✓ The hint appears inline below the input — no modal, no separate request. It arrives in the 400 attempt response and appears immediately.
LevelContent

Creators write level story content in Markdown. react-markdown with remark-gfm supports tables, strikethrough, task lists — useful for ARG-style content with structured data.

Code blocks inside level content get full syntax highlighting via Prism. Creators can embed formatted ciphertext, hex dumps, or code puzzles directly in the story markdown.

External links automatically open in a new tab. The check is simple: if href starts with "http", add target="_blank" and rel="noopener". Internal links (relative paths) navigate normally.

XPToast

The one-frame delay before setting visible: true is essential. If you set visible immediately on mount, the browser has no time to paint the initial state — the CSS transition never fires and the toast just appears instantly.

The toast stacks three layers: XP amount (always shown), rank-up banner (conditional), and badge chips (0 to many). Each is independently conditional so the layout compresses gracefully when absent.

Clicking anywhere on the toast dismisses it early. The 4-second auto-dismiss timer is cleaned up in the useEffect return — no memory leaks if the component unmounts before the timer fires.

useAttempt Hook

Separating fetch logic from the component keeps PuzzleBox focused on rendering. The hook owns all loading state and error handling so the component just calls submit() and reacts to the result.

HTTP 400 (wrong answer) is NOT thrown as an error. It's a normal API response. The hook returns it as data so PuzzleBox can show the wrong-answer UI. Only truly unexpected errors (401, 429, 5xx) throw.
✓ useCallback([puzzleId]) ensures the submit function reference is stable between renders. Safe to put in useEffect dependency arrays in parent components.
Edge Middleware

Runs at the CDN edge before the request hits your server — zero cold-start overhead. Unauthenticated users are redirected to /auth/signin without ever touching the Next.js runtime.

The matcher pattern /create/:path* protects the entire create directory including sub-routes. Adding a new protected page only requires adding a matcher entry here.

The authorized callback checks for a token (session token). NextAuth sets this automatically for authenticated users. No custom logic needed for basic auth protection.

Next.js Config

Image domains are allowlisted so next/image can optimize and serve avatars from GitHub, Google, and Cloudinary. Requests from non-listed domains are blocked by Next.js.

The /thread/:slug → /hole/:slug redirect is a permanent (308) redirect. If you ever rename the URL structure, old links bookmarked by users will continue to work.

Security headers on API routes add a second layer of XSS/clickjacking protection beyond what the framework provides. These headers cost nothing and block entire classes of attacks.

Tailwind Config

Design tokens (bg, surface, accent) are registered as Tailwind colors so you can write bg-bg, text-accent etc. in JSX — no inline style required for brand colors.

The typography theme extension overrides the default prose colors to match the dark theme. Without this, @tailwindcss/typography would render grey-on-white markdown on the dark background.

The shake keyframe (used by PuzzleBox on wrong answers) and slideUp (XPToast) are defined here so they're available as Tailwind utility classes: animate-shake, animate-slideUp.
